# spec.md — Leoni B2B Competency Labeling Landing

## 1) Цель страницы
Одностраничный landing для B2B-партнёров, который ясно объясняет сервис: партнёр передаёт датасет контента (чаще по API), Leoni размечает каждую единицу контента по развиваемым профессионально важным компетенциям на базе внутреннего классификатора и возвращает размеченный датасет обратно (по API), чтобы партнёр мог строить рекомендательные системы и персонализацию.

**Primary CTA:** “Запросить демо / доступ к API”
**Secondary CTA:** “Получить пример разметки”

## 2) ЦА и сценарии
**ЦА:** EdTech, корпоративное обучение (L&D), образовательные платформы, агрегаторы курсов/контента, HR-tech.  
**Сценарии:**
- Понять ценность разметки по компетенциям и как это помогает рекомендациям
- Увидеть простой процесс интеграции (Receive → Label → Return)
- Оценить форматы, каналы передачи, безопасность и SLA
- Оставить заявку на демо / пилот

## 3) Ключевое сообщение (copy ядро)
**Headline:** “Размечаем ваш образовательный контент по развиваемым компетенциям — готово для рекомендаций”  
**Subheadline:** “Получаем ваш датасет по API или альтернативным каналом, размечаем каждую единицу контента внутренним классификатором профессионально важных компетенций и возвращаем результат обратно — тоже по API.”  
**Value bullets (3–5):**
- Разметка каждого объекта контента по развиваемым компетенциям
- Понятный интеграционный контур (вход → разметка → выход)
- Масштабируемость под большие каталоги
- Единая “семантика компетенций” для рекомендаций и аналитики

## 4) Информационная архитектура (блоки)
Порядок блоков фиксирован, допускаются мелкие перестановки для A/B.

### 4.1 Hero
- Заголовок + подзаголовок
- 2 CTA кнопки (primary/secondary)
- Мини-визуал: схема Receive → Label → Return (иконки или линейный шагпер)
- Доверие (опционально): “Для EdTech / L&D / HR-tech”

**UI элементы:**
- Primary button: “Запросить демо / API”
- Secondary button: “Пример разметки”

### 4.2 Как это работает (3 шага)
**Step 1 — Receive:** согласовываем структуру, получаем датасет (API / S3 / файлы / выгрузка)  
**Step 2 — Label:** размечаем единицы контента по компетенциям, которые контент потенциально развивает при просмотре/прочтении/посещении  
**Step 3 — Return:** отдаём размеченный датасет партнёру по API (или оговоренному каналу)

Формат: 3 карточки + короткий текст (1–2 строки каждая).

### 4.3 Что получает партнёр (outcomes)
3–6 карточек:
- Готовый размеченный датасет “контент → компетенции”
- Основа для рекомендательных систем и персонализации
- Повышение релевантности подборок
- Аналитика по каталогу (какие компетенции покрывает контент)
- Возможность поддерживать обновления (инкрементальная разметка)

### 4.4 Интеграции и форматы
Коротко и конкретно.
- Входные каналы: API (основной), файлы (CSV/JSON), S3/облако, secure transfer (по согласованию)
- Выход: API + экспорт (по запросу)
- Модель данных: “content_item” с id, title, description, metadata, url

Примечание: это маркетинговые примеры; реальные поля и договор API уточняются на пресейле.

### 4.5 Безопасность и SLA
- Данные партнёра используются только для выполнения разметки
- Каналы передачи: защищённые протоколы, доступ по ключам/токенам
- Логи/аудит (по договорённости)
- SLA (плейсхолдеры): время обработки, поддержка, окна обслуживания

### 4.6 FAQ (5–7 вопросов)
Рекомендуемые вопросы:
- Чем “компетенции” отличаются от тегов/тем?
- Можно ли разметить разные типы контента (видео/статьи/мероприятия)?
- Как часто можно обновлять разметку и добавлять новые объекты?
- Какие форматы передачи поддерживаются кроме API?
- Можно ли получить пример разметки перед пилотом?
- Как решается вопрос приватности/безопасности?

### 4.7 Финальный CTA + форма
- Повтор заголовка + 1 фраза ценности
- Форма: имя, компания, email, “объём каталога” (опционально), сообщение
- Альтернатива: кнопка “Связаться” + email/телеграм (если есть)

## 5) Требования к UI/UX
- Responsive: 360/768/1024/1440
- Быстрая загрузка: LCP < 2.5s (целевое)
- Чёткая типографика, много воздуха, 2–3 акцентных цвета
- Схема процесса (3 шага) визуально заметна уже на первом экране
- CTA присутствует в Hero и в конце, плюс “липкая” кнопка (опционально на мобиле)

## 6) Технические требования
- Stack: Next.js/React (или другой agreed), TypeScript (желательно)
- SEO: title/description, OG-теги, sitemap/robots, semantic HTML
- Аналитика: события CTA (click_primary_cta, submit_form, click_sample)
- Форму отправлять в: placeholder endpoint /api/lead (добавить мок), или интеграция с CRM (опционально позже)
- Доступность: контраст, aria-label для кнопок, tab navigation

## 7) Контент-ассеты (плейсхолдеры)
- Лого Leoni
- Иллюстрация/иконки для 3 шагов
- (Опционально) 1 схема “content → competencies → recommender”

## 8) Acceptance Criteria (готово, если…)
- Страница содержит все блоки 4.1–4.7 и работает на мобиле/десктопе
- Есть минимум 2 CTA и они ведут к форме/модалке
- “Как это работает” чётко отражает Receive → Label → Return
- Есть блок Integrations & Formats с примерами payload/result (как выше)
- Отправка формы работает на мок-эндпоинт и логирует успех/ошибку
- SEO мета-теги заполнены, OG-картинка подключена (placeholder допустим)

## 9) Нефункциональные ограничения
- Никаких чувствительных деталей классификатора в публичном тексте
- Все данные и SLA — описательно, без жёстких цифр, если не согласованы
